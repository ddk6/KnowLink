## 二、MinerU 文档解析

### 2.1 为什么要换掉 Apache Tika

最初 PaiSmart 使用 Apache Tika 解析上传的 PDF 文件。Tika 的优点是部署简单、无需外部服务，但它的输出是**纯文本**——PDF 里的表格会被拍平成无结构的文字，原本"第一列是日期、第二列是金额"的关系完全丢失，变成一堆数字堆在一起，后续向量检索根本无法理解表格内容。

同样的问题也出现在列表上。"以下情况不在承保范围：（1）核辐射 （2）战争"这样的内容，Tika 只输出一段文字，既不知道哪句是前导说明，哪些是并列的列表项。在切分阶段就更无从判断了。

更关键的是，**没有结构就没有语义边界**。切分时只能按字符数截断，一个 512 字符的 chunk 可能截断在一个句子中间、甚至一个表格数据行的中间，这样的 chunk 用于 RAG 召回效果很差。

### 2.2 MinerU 是什么，为什么选它

MinerU 是一个专业的 PDF 解析服务（官网 mineru.net），它的核心能力是把 PDF 解析成结构化的内容，返回两个重要文件：

- **`full.md`**：完整的 Markdown 格式文本，标题、列表、段落结构清晰
- **`content_list_v2.json`**：逐页逐块的结构化 JSON，每个内容块都有 `type` 字段（paragraph / title / table / list / equation_interline），表格还附带 HTML 格式的内容

有了这个 JSON，我们就知道：这一块是"表格"，那一块是"带前导句的列表"，下一块是"标题"。这样才能做真正有意义的语义切分。

**为什么不选其他方案**：

- **DeepDoc（Python+ONNX）**：需要额外部署一个 Python 微服务，再维护 ONNX 模型文件，运维负担太重。
- **阿里云文档解析**：按量计费且贵，MinerU 已经够用。
- **自研解析模型**：成本和周期都不划算，当前阶段没有必要。

MinerU 选用 HTTP API 方式，整个 Java 后端不需要引入任何新依赖，调用成本低，失败时还可以降级回 Tika。

### 2.3 MinerU 的调用流程

MinerU 不是同步解析的——你上传文件后，它在后台处理，你需要轮询才能拿到结果。整个流程分四步：

**第一步：申请上传链接**

调用 `/api/v4/file-urls/batch` 接口，传入文件名、解析参数（是否开启 OCR、表格识别、公式识别等），MinerU 返回一个预签名的 OSS 上传地址和一个 `batch_id`。

**第二步：上传文件到 MinerU OSS**

用 HTTP PUT 请求，把 PDF 文件的字节流直接推送到那个预签名地址。注意这步跟 MinerU 的 API Server 无关，是直接上传到他们的对象存储，所以超时时间要设得宽松一些（代码里设的是 10 分钟）。

**第三步：轮询等待结果**

反复调用 `/api/v4/extract-results/batch/{batchId}`，检查 `state` 字段。状态会从 `waiting-file` 变成 `processing`，最终变成 `done` 或 `failed`。实现里最多轮询 100 次，每次间隔 5 秒，也就是最多等约 8 分钟。拿到 `done` 状态时，响应里会有一个 `full_zip_url`。

**第四步：下载 ZIP、解压解析**

下载这个 ZIP 包，解压出 `full.md` 和 `content_list_v2.json`（`layout.json` 和原始 PDF 等其他文件直接丢弃，没有用处）。然后用 V3 切分逻辑处理这两个文件。

整个流程都在 `FileProcessingConsumer`（Kafka 消费者）里触发，不阻塞用户的上传请求。如果 MinerU 任何一步失败（超时、返回 `failed`、网络错误），Consumer 会捕获异常并自动降级调用 Tika，用户无感知。

### 2.4 解析结果的存储策略

`full.md` 和 `content_list_v2.json` 存入 MySQL 的 `mineru_parse_result` 表（通过 `file_md5` 关联 `file_upload` 表）。文本内容切分后的 chunk 向量化存入 Elasticsearch。原始 PDF 始终保留在 MinIO 中，不删除，原因是：

1. MinerU 解析失败时需要重试，必须有原始文件
2. 未来模型升级、需要重新解析时，原始文件是唯一来源
3. 合规审计要求原始文档存档

## 三、文档切块策略 V3

### 3.1 原来是怎么切的，问题在哪里

在引入 MinerU 之前，切分逻辑是这样的：把 Tika 输出的纯文本按 `\n\n` 分成段落，每个段落不超过 512 字符；超过的话再按句子切；句子还是太长就用 HanLP 分词。

这套逻辑有几个根本性的问题：

1. **不知道什么是表格**：表格被拍平成了文字，按 512 字符截断会把一张表切成好几个没有上下文的碎片。
2. **不知道什么是列表**："前导句 + 多个列表项"是一个语义整体，但 512 字符的限制会把前导句和列表项切开，导致检索回来的 chunk 只有几条孤立的列表项，没有任何说明。
3. **chunk 太小**：512 字符对于中文来说大约是 250-300 个汉字，对于完整表达一个概念来说太短，很多时候一个完整的说明段落要 800-1000 字符，强行切断破坏了语义完整性。
4. **无法利用 MinerU 结构信息**：即使有了 `content_list_v2.json`，原来的切分逻辑也只用了 `full.md`，白白浪费了结构化信息。

### 3.2 V3 切分的核心思路

V3 的核心原则是：**不同类型的内容，用不同的切分策略**。不再用统一的字符数截断，而是先识别内容类型，再根据类型决定如何切。

`content_list_v2.json` 的结构是一个二维数组，外层按 PDF 页面排列，内层是每一页的内容块。每个块的结构大致如下：

```
{
  "type": "title",
  "content": { "title_content": [{"type": "text", "content": "第3条 保险责任"}], "level": 1 }
}
```

```
{
  "type": "table",
  "content": { "html": "<table>...</table>", "table_caption": [{"type": "text", "content": "财务数据"}] }
}
```

```
{
  "type": "list",
  "lead": "以下情况不在承保范围：",
  "content": { "list_items": [{"content": "(1) 核辐射"}, {"content": "(2) 战争"}] }
}
```

代码在 `MinerUService.parseMarkdownWithStructure()` 里遍历这个结构，根据每个块的 `type` 分派到不同的处理逻辑。

### 3.3 普通文本和标题：section 合并 + 递归细切

对于 `paragraph` 和 `title` 类型，处理逻辑是这样的：

遇到标题时，先把之前积累的内容保存为一个或多个 chunk，然后记下当前标题作为新 section 的名字（用 `>` 分隔构建层级路径，比如 `第一章 产品介绍 > 1.1 主要功能`）。遇到段落时，把它追加到当前 section 的内容缓冲区。

等下一个标题出现时，把整个 section 内容送入 `splitSectionText()` 处理。这个函数的逻辑是：

- 如果内容 ≤ 1024 token，直接作为一个 chunk，不切割。
- 如果内容 > 1024 token 且内部有子标题，按子标题切分，每个子部分再递归处理。
- 如果内容 > 1024 token 且没有子标题，进入句子级切分 (`sentenceAwareSplit`)，按中文句号、感叹号、问号、分号等标点切分，保证不在句子中间截断。

关键条款（标题中含有"保险责任"、"责任免除"、"赔付"等关键词）的 token 上限提高到 1536，因为这类内容密度高，截断后语义损失更大。

### 3.4 表格：HTML 先转结构化文本，再按行切分

表格的处理是最复杂的部分。`content_list_v2.json` 里的表格内容是 HTML 格式（`<table><tr><td>...</td></tr></table>`），直接把 HTML 送入 Embedding 模型有很大问题：`<td>`、`</td>` 等标签本身被当成语义内容处理，产生大量噪声，向量质量极差。

因此第一步是把 HTML 表格转换成结构化文本。`convertHtmlTableToText()` 用正则表达式提取所有 `<tr>` 行和 `<td>/<th>` 单元格，清理掉所有 HTML 标签，然后用 `|` 把单元格拼在一行，得到这样的格式：

```
【财务数据】
指标名 | 2022 | 2023E | 2024E
营业收入 | 3.2亿 | 4.1亿 | 5.0亿
净利润 | 0.8亿 | 1.1亿 | 1.4亿
```

这样的文本 Embedding 效果好得多，模型能识别出"营业收入"和"3.2亿"是对应关系。

转换完成后，估算 token 数：

- **≤ 300 token** 的小表格：整体作为一个 chunk。
- **> 300 token** 的大表格：取前两行作为表头，然后把剩余的数据行按 300 token 的限制切成多段，每段都带上表头。这样每个 chunk 都是可以独立理解的"表头 + 若干数据行"，不会出现没有表头的数据行孤立存在的情况。

如果表格没有 `caption`，用所在 section 的路径作为语义锚点，避免 chunk 完全缺乏上下文。

### 3.5 列表：前导句绑定

列表块（`type: "list"`）的处理逻辑基于一个观察：**前导句是列表的语义说明，列表项是前导句的展开**，两者必须绑定在同一个 chunk 里，不能分开。

`content_list_v2.json` 里 `lead` 字段存的就是前导句（在 block 级别而非 content 级别）。处理步骤：

1. 拼接"前导句 + 所有列表项"，估算总 token 数。
2. 如果总量 ≤ 1024 token，直接作为一个整体 chunk。
3. 如果超出，从列表项逐个累加，超过限制时输出一个 chunk，**每个 chunk 都保留前导句**，然后继续。这样即使一个列表被切成了两段，每段都有"以下情况不在承保范围"这个前导句，检索时才不会出现"没有说明的几条列表项"。

### 3.6 相邻 chunk 的链式结构

所有 chunk 最终会被串联：每个 chunk 有 `prevChunkId` 和 `nextChunkId` 字段，指向前后相邻的 chunk。这为未来实现"扩展召回"（检索到 chunk N 后，同时返回 chunk N-1 和 N+1 的内容）打下了基础，当前阶段这两个字段存入 ES 但暂未启用扩展逻辑。

## 四、查询优化

### 4.1 为什么要做查询改写

原始的搜索流程是把用户输入的文字直接送进 Embedding 和 BM25 搜索。这有几个明显的问题：

- 用户经常输入口语化的疑问句，比如"深度学习框架到底是啥样的啊"。这句话里的"到底是"、"啥样的"、"啊"对检索完全没有帮助，但 BM25 的关键词匹配会受到这些词干扰，反而召回一些词频不相关的文档。
- 全角字符问题：用中文输入法打的"ＡＢＣ"实际上是 Unicode 全角字符（\uFF21-\uFF3A），跟文档里的半角"ABC"不是同一个字符，BM25 自然无法匹配。
- 同义词问题：用户搜"电脑"，文档里写的是"计算机"，两者向量相似度尚可，但 BM25 完全命中不了。

**为什么不用 LLM 改写**：最直接的解法是让 LLM 把用户问题改写成更适合检索的形式（比如 HyDE——让 LLM 先生成一个假设性的答案文档，用那个文档的向量去搜索）。但 HyDE 有幻觉风险：LLM 生成的假设文档如果偏差了，会把向量拉偏，反而降低召回质量。而且每次查询都要额外调一次 LLM，增加了几百毫秒的延迟。规则方法零延迟、可预测、可测试，对当前场景完全够用。

### 4.2 查询改写的四个步骤

**第一步：全角转半角**

全角字符范围是 `\uFF01-\uFF5E`，转换公式很简单：`半角 = 全角 - 0xFEE0`。全角空格（`\u3000`）单独处理转成普通空格。这一步在其他任何处理之前做，确保后续步骤看到的都是标准化字符。

**第二步：移除特殊字符**

用正则表达式把冒号、斜杠、引号、括号等对语义检索没有贡献的特殊字符替换成空格，合并多余空白。保留了中文句号、逗号等语义标点，不做处理。

**第三步：WWW 疑问词过滤**

这步的名字来自"What/Where/Why/Who/How"（WWW），但实际上覆盖了中英文的疑问词。处理逻辑是：

- 移除**前缀**疑问词：如"请问"、"什么是"、"如何"、"哪里"等，这些词出现在句首时通常不携带检索语义。
- 移除**后缀**疑问词：如"吗"、"呢"、"吧"、"啊"，这些语气词出现在句尾对检索毫无帮助。
- 移除英文疑问词：如"what"、"how"、"is"、"are"等常见的英文功能词。

例如："深度学习框架到底是啥样的啊" → 移除后缀"啊"和"啥样的" → "深度学习框架到底是" → 再移除"是" → "深度学习框架"。

**第四步：同义词扩展**

在 `resources/dict/synonym.txt` 里维护一张同义词表，每行格式是"主词,同义词1,同义词2"。查询时对清理后的词做哈希查表，匹配到同义词后把它们追加到原始查询后面，用空格分隔，最多追加 5 个同义词，避免 Query 过长。最终结果例如：`"深度学习框架 神经网络 机器学习框架"`。BM25 对这个扩展查询用 `AND` 操作符——等等，实际上当前实现是直接拼在查询字符串后面，BM25 用 `match` 查询会按 OR 语义处理扩展词，保证召回不缩窄。

如果 `synonym.txt` 文件不存在，系统会加载一批内置的默认同义词（AI/机器学习/深度学习等技术领域词汇），保证服务始终可用。

### 4.3 RRF 融合召回

原来的搜索只用 KNN（向量相似度）召回，对精确关键词的支持很弱。改造后同时做两路召回：

- **KNN 召回**：对用户查询做 Embedding，在 Elasticsearch 的 `knowledge_base` 索引里做近邻搜索，返回语义相似的文档列表（按相似度排名）。
- **BM25 召回**：对查询字符串做标准的全文检索，返回关键词匹配的文档列表（按 BM25 分数排名）。

然后用 **RRF（Reciprocal Rank Fusion）** 把两路结果融合。RRF 的思路是：不比分数绝对值，只比排名。公式是：

```
RRF_score(doc) = knnWeight / (k + rank_knn) + bm25Weight / (k + rank_bm25)
```

`k` 通常取 60，作用是平滑掉排名靠后的文档之间的差异。如果一个文档只在 KNN 里出现、没有在 BM25 里出现，那它的 BM25 贡献为 0，不会被惩罚，但也不会额外加分。

这样的设计让两路召回互补：语义相近但关键词不同的文档靠 KNN 召回，精确匹配关键词的文档靠 BM25 召回，最终融合结果覆盖两种情况。

整个召回阶段会扩大 K 值（`topK * 30`），尽量多捞文档，为后续的重排序提供充足的候选。

### 4.4 Cross-Encoder 重排序

RRF 的融合结果取前 20 条候选文档，送入 Cross-Encoder 重排序（`qwen3-rerank` 模型，调用阿里云 DashScope 的 Rerank API）。

**为什么还需要重排序**：RRF 是一种"排名融合"，它的得分只反映文档在两路召回里的相对排名，无法真正衡量文档内容和用户查询的相关程度。Cross-Encoder 会把"用户查询 + 文档内容"一起输入模型，让模型计算真实的相关性分数，排序质量明显更高。代价是每次查询需要额外调一次 API（约 200-500ms），但这是可以接受的。

如果 Rerank API 调用失败，系统自动降级到 RRF 融合结果直接返回。如果向量生成也失败，继续降级到纯 BM25 文本搜索。整个链路有三级降级保障，不会因为某个服务不可用而完全失效。

## 五、对话记忆模块

### 5.1 问题背景

最初 PaiSmart 的对话历史存储在 Redis 里，格式是一个 JSON 字符串（整个历史是一个 JSON 数组），保留最多 20 条消息。这个方案有两个问题：

第一，**旧数据与新代码不兼容**。后来改成用 Redis List（每条消息独立存一个 JSON），但老会话的数据还是 String 格式。新代码用 LRANGE 操作 String 类型的 key 会报 `WRONGTYPE` 错误，导致老会话数据全部丢失。

第二，**只有短期记忆，没有长期积累**。用户在某次对话里告诉了系统"我是做保险精算的"，5 轮之后这条信息就滚出窗口消失了。下次新开一个会话，系统完全不记得这位用户的背景。这对个性化推荐和精准回答很不利。

### 5.2 短期记忆：Redis List

短期记忆负责维持当前对话的上下文，让 LLM 知道"我们刚才聊了什么"。

**存储格式**：Redis List，key 是 `conversation:{conversationId}`。每条消息独立序列化成 JSON 字符串存入 List，而不是整个数组序列化成一个 String。这样可以用 `RPUSH` 追加、`LRANGE` 取范围、`LTRIM` 截断，操作更高效，也不会出现对大 JSON 的整体读写。

**窗口大小**：保留最近 5 轮对话，即最多 10 条消息（5 条 user + 5 条 assistant）。每次追加完新消息后，如果 List 长度超过 10，就 `LTRIM` 掉最旧的消息。TTL 设为 7 天，超期自动失效。

**消息格式**：每条消息不只有 `role` 和 `content`，还有 `timestamp`（时间戳）和 `token_estimate`（估算的 token 数），后者为将来可能实现的基于 token 数的窗口策略留了扩展点。

**旧数据迁移**：系统在读取时会先尝试 `LRANGE`，如果 List 为空，再尝试 `GET` 读取旧的 String 格式。读取成功后，删除 String key，然后把解析出的消息重新以 List 格式写入。这样旧会话的历史不会丢失，迁移在第一次读取时自动完成，用户无感知。

### 5.3 长期记忆：Elasticsearch

长期记忆负责跨会话的信息积累，让系统记住"这个用户说过的重要信息"。

**触发时机**：每次向 Redis List 追加完新的两条消息（一问一答）后，检查 List 长度是否已经达到上限（10 条）。如果是，说明有旧消息要被滚出，在 LTRIM 之前，先取出最旧的 2 条消息（即将被滚出的那一轮对话），作为"溢出消息"传给长期记忆模块处理。这样第 5 轮之后的每一轮，都会触发一次对"最旧那一轮"的整合。

**LLM 摘要抽取**：收到溢出消息后，`ConversationMemoryService.consolidateMemory()` 会调用 LLM（使用当前配置的活跃 LLM Provider），传入一个 Prompt，要求 LLM 判断这段对话是否值得长期保留，以及应该归为哪种类型：

- `task`：用户描述的任务或目标
- `preference`：用户的偏好（如"喜欢简短的回答"）
- `fact`：关于用户或领域的客观事实
- `episode`：一段值得记住的具体事件
- `constraint`：用户提出的约束条件（如"不要推荐价格超过X的产品"）

LLM 返回 JSON 格式，包含 `should_store`（是否值得存）、`summary`（一句话总结）、`details`（补充说明）、`entities`（涉及的实体名词）、`keywords`（关键词）、`importance`（0-1 重要性评分）、`confidence`（0-1 置信度）。

**去重与合并**：写入 ES 前先调用 `findSimilarMemory()`，在同一用户的已有记忆里用 BM25 match 查询找是否存在相似的记忆（相同 userId + 相同 memoryType + 摘要内容相似）。如果找到，就把新的 `details` 追加到已有记忆的末尾，同时提升 confidence（`+0.1`，上限 1.0）。如果没有找到相似记忆，就创建一条新记录，TTL 设为 180 天。

**存储位置**：ES 索引 `conversation_long_term_memory`，使用 `ik_max_word` 分词器索引 `summary`、`details`、`entities` 字段，支持中文全文检索。每条记忆还有一个 `summaryEmbedding` 字段（dense vector，1024 维），当前存入但未使用，为未来向量检索预留。

### 5.4 长期记忆如何参与对话

每次用户发来新消息时，在执行 RAG 检索之前，ChatHandler 会先调用 `conversationMemoryService.retrieveMemories()` 检索长期记忆。检索条件是：`userId` 匹配 + `isActive = true`，用用户输入做 BM25 match 查询，返回最相关的 5 条记忆。

检索到的记忆会被格式化成一段文本，格式如下：

```
【长期记忆】
[记忆1-preference] 用户偏好简短、直接的回答方式
[记忆2-fact] 用户从事保险精算工作，关注责任免除条款
```

这段文本会拼接在 RAG 检索到的文档上下文**前面**，一起送给 LLM 作为参考。这样 LLM 在回答时既有文档知识，又能记住这个用户的个人背景。

### 5.5 已知局限与待优化项

当前的长期记忆检索只用了 BM25 关键词检索，对语义相近但关键词不同的记忆召回能力有限。`summaryEmbedding` 字段已经在 ES Mapping 里定义了 dense vector，等向量化后可以改为 KNN + BM25 混合检索，效果会更好。

`buildRetrievalQuery()` 目前只是简单地把用户输入原样作为检索 Query，没有做任何扩展或改写。如果用户的问题和记忆里的关键词差异较大（如用"电脑"查询，但记忆里写的是"计算机"），可能召回不到相关记忆。

此外，过期记忆的清理目前只靠 `is_active = false` 软删除，没有定时清理任务，ES 里的无效数据会逐渐积累。
